## 本集核心地圖（兩條軸）

連接外部工具，先看**兩個問題**：①是不是 **Google 家**的服務？ ②你要「開箱即用」還是「完整控制」？

|  | 非 Google | Google 家 |
|---|---|---|
| **簡單路（開箱即用）** | 內建連接器 / MCP ＋ API Key | 原生連接器（Drive／Gmail 開關即用） |
| **完整控制路** | MCP ＋ OAuth / 本地直存 | 自建 GCP 專案 ＋ OAuth 2.0 |

> 這張 2×2 畫的是**典型路**，Google 欄以 **Workspace 個資服務**（Gmail／行事曆／Drive／Classroom）為代表。**Google 家還有 Firebase、NotebookLM 走不一樣的授權**——完整清單見〈第二大段〉的「Google 家服務總覽」。

### 底層原理：兩條階梯（取代難懂的矩陣）

Google／非 Google 是「入口」，但底層其實是**兩個獨立問題**——「怎麼接」與「用多大權限接」。與其排成一張難懂的網格，用**兩條各有排序的階梯**呈現，一眼可讀。

**核心比喻（片中一定要講）**：**通道是「水管」，授權是「水管裡流的水」。** 你會問「這條 CLI 該配 Token 還是 OAuth」，但**不會問「該用 CLI 還是 Token」**——那是兩件不同的事。

#### 階梯 A：鑰匙（授權）—— 隱性 → 顯性（由鬆到嚴）

| 鑰匙 | 給它多少權 | 收得回嗎 | 白話 |
|---|---|---|---|
| **免鑰匙** | 靠本機權限／你已登入的狀態 | — | 它站在你已經開好的門裡（Obsidian、操控畫面） |
| **API Key／Token** | 一把**萬能長期**鑰匙，貼了就全開 | 要手動撤 | 家裡大門鑰匙（資料庫、Notion） |
| **OAuth** | **限範圍、會過期** | 隨時撤 | 門禁卡（Google Workspace、`gh`、`rclone`） |

> 這條「由鬆到嚴」直接餵給安全收尾：**能用 OAuth 就別到處貼 API Key。**

#### 階梯 B：通道（怎麼接）—— 隱性 → 顯性（由現成到硬幹）

> **兩條階梯同一個方向**：由「**隱性**」（藏起來、看不到怎麼運作）走到「**顯性**」（明確、看得見動作）。通道從連接器（藏起來的插座）走到操控畫面（看它一步步點）；鑰匙從免鑰匙（沒有明確授權）走到 OAuth（明確登入、同意畫面）。

| 通道 | 一句話 |
|---|---|
| **內建連接器** | 原廠接好，開關即用 |
| **MCP** | 萬用轉接頭，社群做一堆 |
| **CLI** | 跑命令列驅動（`gh`／`rclone`），還能自造（CLI-Anything） |
| **直接操控畫面** | 沒 API 也能連，最後手段 |

> **CLI 是第四種通道**：它就是 EP03 教的「跑程式」能力指向外部工具。甚至有工具能**自動幫沒 CLI 的軟體造一個 CLI**（如 CLI-Anything，掃描原始碼自動生成命令列 + `SKILL.md`）——這個「打包成 Skill」的玩法留到 **EP05** 細講。

#### 搭配圖：兩條階梯怎麼接起來（片中放一張接線圖）

看完兩條階梯後，用**一張接線圖**收束——**兩條階梯同向（上隱下顯）並排**，連線按階梯順序連（此圖已畫好 SVG，可直接搬進 slides）：

- **MCP 最靈活**：唯一分岔兩把——接資料庫用 API Key、接本機 Obsidian 免鑰匙。
- **首尾對調的反向規律（片中要點出，圖上用顏色標）**：**最隱性的通道（連接器）配最顯性的鑰匙（OAuth）；最顯性的通道（操控畫面）配最隱性的鑰匙（免鑰匙）**。所以連線在中間交叉成 X——這不是畫錯，是真實規律。
  - 為什麼：連接器藏起來，是因為原廠幫你把 OAuth 接好了；操控畫面之所以要你看它硬點，正是因為**沒有正式授權可用**，只能借你的登入狀態。
- **交叉本身就是證據**：如果通道和鑰匙必然一一對應，線就會是斜對角、不會交叉——會交叉，正好證明**這是兩條獨立的軸**。

> **一句話收束**：**通道決定「怎麼接」，鑰匙決定「給多大權」；兩軸獨立，所以 MCP 能配任何鑰匙、連接器與操控畫面卻首尾對調。**

> **CLI「登入」的原理（口白補充）**：那些 `xx login` 骨架都一樣——**登入一次 → 拿憑證 → 存本機 → 之後自動帶**；主流走正式 OAuth（可撤銷、安全），少數走 cookie 重用（如 NotebookLM 的 nlm，較脆、偶爾要重登）。

> **名詞界定（片中要講清楚，避免誤導）**：
> - 本集**「Google 家」＝所有 Google 擁有的服務**：Gmail／行事曆／Drive／Classroom／NotebookLM／Firebase。用「**誰擁有它**」來分，最直覺、不會混。
> - **但 Google 家內部，授權方式不只一種**：Workspace 個資（Gmail／行事曆／Drive／Classroom）走 **OAuth**；**Firebase** 走自己的**服務金鑰**；**NotebookLM** 走**登入式**（nlm）。詳見下方「Google 家服務總覽」。
> - **Supabase 不是 Google**（是 Firebase 的獨立替代品），歸非 Google。→ 同樣是資料庫，**Firebase 在 Google 家、Supabase 在非 Google，因為擁有者不同、接法就不同**。

## 第一大段：非 Google 系列（約 12 分鐘）

### 簡單路：內建連接器 ＋ MCP ＋ API Key
- **內建連接器**：有些 Agent 原廠就接好了。例：**Codex 在 `Settings → Apps` 有 GitHub 連接器**，登入一次就能讓它讀你的 repo。這類是 OAuth 在底層跑、你只按開關。
- **MCP ＋ API Key**：服務沒內建？只要有人做了 MCP server，貼一組 API Key／Access Token 進設定檔就能接。這是**最原始、最快**的授權方式。
  - 例：資料庫金鑰、Notion token、各種服務的 access token。

### 完整控制路：MCP ＋ OAuth／本地直存
- **OAuth（升級版授權）**：比 API Key 安全——**登入授權一次、範圍受限、可隨時撤銷**。遠端 MCP server（HTTP）現在多支援 OAuth。
- **本地直存（免 token）**：像 Obsidian 讀你本機的筆記庫，根本不需要 token，靠的是**檔案系統權限**。
  - 說明：**免鑰匙本身就是一種正當方式**（階梯 A 的第一階），不是「陽春」。它放在完整控制路，是因為**資料全在你自己電腦、你握有最完整的掌控**——最無痛、也最私密。

## 第二大段：Google 系列（約 14 分鐘）

### 為什麼 Google 要獨立一段？
- 老師世界一大半在 Google（Classroom 發作業、Drive 存教材、Gmail 寄通知），所以這段對老師特別重要。
- 而且 Google 家的服務**接法自成一格**——尤其 Workspace 個資**不給你簡單 API Key 就能存取**，這是跟非 Google 最大的不同。

### Google 家服務總覽（本段核心表）
把老師會用到、我影片也拍過的 Google 家服務一次攤開——**重點：連 Google 自己家，接法都不一樣**：

| 服務 | 老師拿來做什麼 | 接法（通道） | 鑰匙（授權） | 舊影片 |
|------|--------------|-------------|-------------|--------|
| **Gmail** | 自動寄通知／回信 | 原生連接器 ／ 自建 OAuth | OAuth | 觀念帶過 |
| **行事曆** | 排課表／排程 | 原生連接器 ／ 自建 OAuth | OAuth | 觀念帶過 |
| **雲端硬碟** | 讀教材、整理講義、產出簡報 | 原生連接器 ／ 自建 OAuth | OAuth | 現場帶／觀念帶過 |
| **Classroom** | 發作業、抓學生名單、整理繳交狀態 | 自建 OAuth（要寫入） | OAuth | AntiGravity 基本功 EP06 |
| **NotebookLM** | 教材生簡報／音訊 | CLI（nlm） | **登入式** | Claude 基本功 EP03／EP04／EP06 |
| **Firebase** | 即時互動資料庫、課堂遊戲記分 | MCP ／ CLI | **服務金鑰** | Claude 基本功 EP095、GPTcodex 基本功 EP04 |

> **一看就懂的兩群**：上面 4 個是 **Workspace 個資 → 一律 OAuth**；下面 2 個是 **Google 的開發／AI 工具 → 各走各的**（Firebase 用服務金鑰、NotebookLM 用登入式）。下面「四種方式」講的是**上面那 4 個 Workspace 服務**的接法。

### 連接 Google（Workspace 個資）的四種方式（不是單純控制力階梯）
| 方式 | 適合情境 | 說明 |
|------|---------|------|
| **原生連接器** | 偶爾讀取、零設定 | Gemini／AntiGravity 原生、或 Claude／ChatGPT 的 Google Drive 連接器。OAuth 在底層跑，你只按開關；範圍被原廠框住（多半 Drive/Docs 讀取為主） |
| **自建 GCP 專案 ＋ OAuth 2.0** | **要寫入、要細緻 scope、要完整掌控** | 自己開 Google Cloud 專案、啟用要的 API、設 scope、拿憑證。想寫入、想動 Classroom/Gmail，得走這條 |
| **Service Account** | 伺服器自動化、共用資料、網域委派 | 伺服器對伺服器、免互動登入；但碰不到你個人 Gmail／Drive（除非 Workspace 網域授權）。適合共用資料、自動化 |
| **API Key** | 公開資料或不碰個資的 Google API | **只能用在不碰個資的 Google API**（Maps、Gemini API 等）。**存取不了 Drive／Gmail／Classroom 的個人資料** |

> **破除迷思（片中一定要講）**：很多人以為「Google 給我一組 API Key 就能讀我的 Classroom」——**不行**。個人資料一律要 OAuth 或 Service Account。

### 完整控制路：自建 GCP ＋ OAuth 2.0（只講觀念 ＋ 何時該用，不踏設定步驟）
- **一句話**：要**完整、可控**地讓 Agent 操作你的 Google（尤其「寫」的操作、細緻 scope），自建 GCP 專案走 OAuth 2.0 是最完整的路。
- **何時該用**：
  - ✅ 要**寫入**（Gmail 自動寄信、Classroom 發作業、Drive 建檔）→ 值得自建 OAuth。
  - ✅ 要**細緻控制權限範圍**（只給讀 Calendar、不給刪 Drive）→ 自建 OAuth。
  - ❌ 只是偶爾讀個 Google Doc → 原生連接器就夠，別自找麻煩。
- **代價（點到為止）**：設定較繁——開 GCP 專案、分別啟用各 API、設同意畫面、拿憑證、處理 token 更新。**細部步驟不在本集，之後做懶人包。**
- **取捨口訣**：**原生連接器＝零設定但受限；自建 OAuth＝要花工但完全掌控。**

### 舊影片回扣（Google）
- 舊影片對照已併入上方「Google 家服務總覽」的最後一欄：**Classroom** 回扣 AntiGravity 基本功 EP06；**NotebookLM** 回扣 Claude 基本功 EP03／EP04／EP06；**Firebase** 回扣 Claude 基本功 EP095、GPTcodex 基本功 EP04；Drive／Gmail 的原生連接器現場帶、OAuth 觀念帶過。

## 第三段：直接操控畫面（Computer / Browser Use）—— 口頭帶過（約 4 分鐘）

- 兩邊都沒 API、沒 MCP、沒連接器怎麼辦？**讓 Agent 直接操作畫面**，像真人點滑鼠、打字（Computer Use／瀏覽器操控／Chrome 外掛）。授權靠的是**你已登入的 session**，不用 token。
- 特點：**萬用但脆**——畫面一改就可能失手、速度慢。是**最後手段**，不是首選。
- 一句話：**能用連接器或 MCP，就別讓它硬點畫面。**

## 收尾 1：三步選路法（必講，約 4 分鐘）

一張決策流程收束整集：

1. **是 Google Workspace 的個資服務嗎？**
   - 是 → 只是讀 → **原生連接器**；要寫入／要控制 → **自建 OAuth**。
   - 不是 → 往下。
2. **有內建連接器或現成 MCP 嗎？**
   - 有 → **用它**（貼 API Key，或升級成 OAuth 更安全）。
   - 沒有 → 往下。
3. **連 API 都沒有** → **直接操控畫面**（最後手段）。

> **總口訣**：**先問是不是 Google，再問要不要完整控制。**

## 收尾 2：安全提醒（必講，回扣 EP02／EP03）

- 這集讓 Agent 的手腳伸出電腦——**能碰到雲端、Classroom、資料庫，風險也放大了**。
- 三個原則：
  1. **看清楚你交出去的是哪把鑰匙**：API Key 是「萬能長期鑰匙」，OAuth 是「範圍受限、可撤銷」——**能用 OAuth 就別到處貼 API Key**。
  2. **學生資料走去識別化**：接 Classroom／資料庫尤其要注意，只存座號＋班級代號。（細節後面集數）
  3. **來路不明的 MCP server / 連接器不要亂裝**：它會拿到你的授權，跟裝來路不明外掛一樣要小心。
- 操作口訣：**不用的 token 要撤、API Key 要限制用途、OAuth scope 只開需要的。**
- 一句話：**「接得越多，越要看清楚你把哪把鑰匙交出去了。」**

| 操控畫面 | 能用連接器或 MCP，就別讓它硬點畫面 |
| 安全 | 接得越多，越要看清楚你把哪把鑰匙交出去了 |
| 新手起手式 | 別想著一次接全部——挑一個你今天就會用到的，先接起來 |
| 下集鉤子 | 這集給你工具，下集幫你把工具用法變成一句話 |
